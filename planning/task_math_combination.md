# `math/combination.hpp` 移植タスク仕様案

## 1. 概要
順列・組合せ（階乗、階乗の逆元、二項係数など）を高速に計算するためのモジュールを移植・再設計します。
元の `master` ブランチでは、関数テンプレート内で `static` なテーブル（`combination_table<T>`）を保持し、必要に応じて自動でサイズを倍加拡張（`extend`）する設計になっていました。

## 2. 旧設計の問題点（DynamicModInt との相性）
元の設計（静的グローバルなテーブルキャッシュ）には、競技プログラミングにおいて致命的なバグを引き起こす可能性が潜んでいます。
- 今回のプロジェクトでは、実行時に法が変更できる `DynamicModInt64` を導入しています。
- もし `static combination_table<DynamicModInt64>` としてキャッシュを持った場合、ユーザーが `set_mod()` で法を変更してもテーブルの内容は古い法のまま保持されてしまい、計算結果が壊れてしまいます。
- これを防ぐためには「法が変わったかを検知してテーブルをリセットする」機能が必要ですが、毎回チェックするのはオーバーヘッドになります。

## 3. 提案する新設計
上記の問題を根本から解決し、かつスレッドセーフにするため、**明示的なクラス（`class Combination<T>`）** として再設計することを提案します。

```cpp
template <gwen::modint T>
class Combination {
    std::vector<T> F, I;

public:
    // コンストラクタで最大 N まで前計算を行う
    explicit Combination(int n = 0);

    // 必要なサイズまで動的にテーブルを拡張する
    void extend(int n);

    // 階乗 n! (内部テーブルを拡張する可能性があるため非const)
    T fact(int n);

    // 階乗の逆元 1 / (n!) (内部テーブルを拡張する可能性があるため非const)
    T fact_inv(int n);

    // 二項係数 nCk (内部テーブルを拡張する可能性があるため非const)
    T comb(int n, int k);

    // 順列 nPk (内部テーブルを拡張する可能性があるため非const)
    T perm(int n, int k);
};
```

### 設計のポイント
- **`modint` コンセプトによる制約**: テンプレート型 `T` には `gwen::modint` コンセプトを満たすことを要求します。これにより、逆元や除算が確実に提供されることが保証され、整数など不適切な型によるコンパイルを安全に防ぐことができます。
- **非 `const` メソッド**: 自動でテーブルを拡張（`extend`）する際、内部の `std::vector` を書き換えるため、メソッドはあえて非 `const` とします。`mutable` を用いると複雑化や思わぬ副作用（スレッド安全性など）を招くため、競プロの慣習的にも非 `const` メソッドでの提供が最も素直で安全です。
- **オブジェクトベース**: クラスとしてインスタンス化することで、法が異なるテストケースでもスコープごとに `Combination` オブジェクトを作り直すだけで安全に動作します。
- **動的拡張**: 元の実装の長所である「自動で倍々拡張する」機能は残し、`fact(n)` などが呼ばれた際に `n` が現在のサイズを超えていれば自動的に `extend(n)` を呼ぶようにします。これにより、初期化時に正確な $N$ を見積もる必要がなくなります。

## 4. 計算量
- **テーブル初期化 / 拡張**: $O(N)$
- **`fact`, `fact_inv`, `comb`, `perm`**: 全て $O(1)$ （ならし $O(1)$）
