# Dump機能 実装・保守ルール

本ファイルは、gwenプロジェクトにおける `dump` 機能の設計および実装に関するルールをまとめたものです。
AIエージェントが新規モジュールを実装したり、既存モジュールを改修したりする際に参照してください。

## 1. 設計思想（シンプルさ優先）
- `dump` 機能は**ローカル環境でのデバッグ用途**を前提としています。
- パフォーマンス（O(1)やO(log N)での取得など）よりも、**内部実装のシンプルさと保守性の高さ**を優先してください。
- 特に、遅延評価セグメント木などのデータ構造において、正確な値を副作用なし（`const`）で取得するためにディープコピーや全要素の走査（O(N)）が必要になる場合は、それを許容します。複雑なクエリベースの取得処理を `dump` のためだけに実装することは避けてください。

## 2. DUMP_SIZE_LIMITの対応
巨大な配列やデータ構造をダンプする際、コンソール出力が長くなりすぎるのを防ぐため、`DUMP_SIZE_LIMIT` に対応してください。

- 出力時の文字列フォーマットの段階で要素を切り詰めます。`to_vec()` のようなメソッドで一旦全要素を取得し、その戻り値に対して制限を適用するアプローチが推奨されます。
- 出力フォーマット: `[elem1, elem2, ..., elemN, ... , last_elem]` 
  - `N = DUMP_SIZE_LIMIT` とする。
  - C++23 の `std::views::take` や `std::ranges` を活用して安全に切り詰めること。

## 3. 各データ構造の出力要素
データ構造モジュールの `dump()` メソッドは、内部の生データだけでなく、**ユーザーから見て論理的に意味のある状態** を出力してください。

*   **FenwickTree (BIT)**: `N`, `data(restored)` (元の論理配列)
*   **SegmentTree / LazySegmentTree**: `N`, `data(restored)` (葉の要素), `lazy` (溜まっている遅延配列)
*   **DynamicSegmentTree**: `N`, `size`, `elements` (有効なインデックスと値のペア一覧)
*   **UnionFind / DSU**: `N`, `groups` (連結成分ごとの頂点リスト群)
*   **Graph / Digraph**: `V`, `E`, `adjacency_list` (制限適用)
*   **Tree (木構造)**: `V`, `root`, `parents`, `depths`
*   **Zobrist Hash**: 登録されている `(key, hash_value)` の一覧 (制限適用)
