# gwen プロジェクト指示書 (GEMINI.md)

このファイルは、AIエージェント（Antigravity）が本プロジェクト「gwen」で動作する際の開発・行動指針を定義したものです。

---

## 1. 開発方針・ディレクトリ構成ルール
* **ヘッダーオンリー**: 本ライブラリはヘッダーオンリーです。実装はすべて `include/gwen/` 配下の適切な名前空間ディレクトリに `.hpp` 形式で作成・追加します。
* **C++23 Concept の利用**: グラフ (Graph) や代数構造 (Monoid, Abel) などの汎用コンポーネントは、C++23 `concept` を用いて厳密な制約を設けます。特殊なものは都度検討します。
* **外部参照の隔離**: 外部のリファレンスコード（例: `nyaan` 等）は、自作コードと混ざらないように `references/` ディレクトリ配下に隔離します。
* **ビルド成果物とテンポラリファイルの管理**: ビルド成果物は `.build/` ディレクトリへ出力してください。また、AIエージェントが動作確認などで一時的なファイル（テストコード等）を作成する場合は、プロジェクトルートを汚さないよう必ず `temp/` ディレクトリ配下に配置してください。これらはGit追跡から完全に除外（`.gitignore`）します。
* **エディタ設定の除外**: `.vscode/` 設定フォルダはローカルのみで保持し、Gitには含めません。

---

## 2. ドキュメントとスクリプトの形式的管理
* **重要ドキュメントの事前参照**: AIエージェントは、新規タスクの着手時や方針に迷った際、プロジェクトの開発手順やコーディング規約が記載された重要ドキュメント（`doc/DEVELOPMENT.md`, `doc/RULES.md` など）を必ず事前に読み込み、その内容に従って実装や提案を行ってください。
* **ドキュメント管理**: `README.md`, `GEMINI.md`, `planning/TODO.md` は形式的に管理します。プロジェクトの状態や規約が変更された場合は、これらのドキュメントを常に最新の状態に保つようにします。
* **スクリプト管理**: `scripts/` に配置するスクリプトは、用途を明確にし、必要に応じて `README.md` 等に使い方を明記して形式的に管理します。すべてターミナル上で完結するようにします。

---

## 3. テスト・検証ルール
* **新規実装時の必須対応**: 新規機能追加やモジュール移植時は、必ず「Google Test による単体テスト」と「competitive-verifier 用の verify コード」の両方をセットで作成し、パスさせます。
* **Google Testの利用**: テストフレームワークには C++ Google Test を使用し、ビルドには CMake を使用します。ユニットテストは `test/unit/` 配下に各モジュール用の gtest コードを作成します。
* **Verifyの実行**: オンラインジャッジ提出用のコードは `verify/` 配下に格納し、`competitive-verifier` を用いて検証を行います。

---

## 4. コミュニケーションと言語ルール
* **言語仕様**: ユーザーへの最終的な回答、コメント、ドキュメントの更新はすべて自然な **日本語** で行ってください。
* **実装計画の優先**: 大きなリファクタリングや機能追加を行う前には、必ず `implementation_plan.md` を作成してユーザーの承認を得るようにプロセスを守ってください。

## 5. コミットメッセージのルール
本プロジェクトでは以下のプレフィックスを付けた緩やかな Conventional Commits を使用してください。
- \feat:\ : 新機能・新しいモジュール
- \fix:\ : バグ修正
- \docs:\ : ドキュメントの更新
- \test:\ : テスト・検証コードの追加
- \chore:\ : CI/CD、その他雑務

## 6. 個別作業のワークフロー
各モジュールや機能単位の個別作業を進める際は、以下の厳格なフローに従ってください。
1. **作業用ファイルの作成**: `planning/` 以下に個別の作業用mdファイル（例: `planning/task_mod_modint.md`）を作成し、1つずつ作業を進めます。
2. **必須ドキュメントの確認**: 作業開始時に必ず必須ドキュメント (`GEMINI.md`, `doc/DEVELOPMENT.md`, `doc/RULES.md`) を読み込みます。
3. **作業ステップ**: 以下の順序でプロセスを回し、各レビュー段階でユーザーの承認を得てください。
   1. **仕様検討**: 作成した作業用mdファイルに仕様案（パス、Conceptの利用方針、依存関係など）をまとめます。
   2. **サブエージェントによる事前レビュー**: サブエージェント（例: Code Reviewer）を呼び出して仕様案の事前レビューを受け、フィードバックがあれば反映します。
   3. **仕様レビュー**: ユーザーに仕様案を提示し承認を得ます。
   4. **実装**: `include/gwen/` 以下に `.hpp` を実装します。テンプレート引数に制約が必要な場合は基本的に C++23 `concept` で縛ります。
   5. **サブエージェントによる事前レビュー**: 実装コードについてサブエージェントにレビューを依頼し、問題点を修正します。
   6. **実装レビュー**: ユーザーに実装コードのレビューを依頼します。
   7. **テスト実装 + テスト実施**: `test/unit/` 以下にテストを書き、パスさせます。Verifyコードが必要な場合は `verify/` にも追加します。
   8. **ドキュメント作成**: `doc/` 以下にドキュメントを作成し、`.verify-helper/docs/index.md` に追加します。
   9. **サブエージェントによる事後レビュー**: テストやドキュメントを含めた全体をサブエージェントに最終確認させます。
   10. **総合レビュー**: 全ての成果物をユーザーに確認してもらいます。
   11. **修正と再レビュー**: 指摘があれば修正し、再度レビューを受けます。
   12. **コミット**: 全て承認されたらコミットします。
